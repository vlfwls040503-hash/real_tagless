## 3-2. 불일치 시나리오 없음 → 대신 '정량적 trade-off' 관찰

최적 배합 자체는 일치하지만, **에스컬 부담의 질적 변화**는 관측됨.

### 게이트-에스컬 시간 trade-off
```
  p  best_cfg  best_gw  best_esc  best_post_ratio
0.1         1    14.18     12.24            50.49
0.3         2    10.61     14.69            56.46
0.5         3     9.04     16.26            59.24
0.7         4     7.19     16.61            61.73
0.8         4    11.62     14.99            56.42
```
- `best_gw`: 최적 cfg의 게이트 대기 (평균초)
- `best_esc`: 최적 cfg의 에스컬 대기 proxy (평균초)
- `best_post_ratio`: 최적 cfg의 post_gate / total 비율 (%)

관찰: p가 커질수록 best_esc도 증가 경향. 특히 p=0.5~0.7에서 게이트 대기 ≈ 에스컬 대기 수준. 이는 병목이 게이트→에스컬로 **부분 전이**됨을 시사.

## 3-3. 통합 결론 (A / B / C)

사전 정의:
- A: 모든 p에서 gate-only = total 최적 → **에스컬 여유, 병목 전이 없음**
- B: 일부 p에서 불일치 → **병목 전이 관측**
- C: 판단 불가 → 데이터 부족

### 결론: **A + 조건부 B** (하이브리드)

**A 측면 (최적 배합 불변)**:
- gate-only, total, pass_rate 3개 기준 모두 동일한 최적 배합 도출.
- 불일치 0건. 에스컬 용량이 최적 선택을 바꿀 만큼 부족하지 않음.

**B 측면 (정량적 전이 증거)**:
- 실측 게이트 처리율 vs 에스컬 대기 proxy: r=+0.48, R²=0.23, p<0.0001.
- 최적 배합에서 post_gate가 total의 55-62% (비최적은 22-33%).
- **게이트를 잘 푸니(빨라지니) 에스컬이 상대 병목으로 떠오름**. 총 시간 감소는 이뤄졌지만 병목 위치가 이동.

### 실무 시사
1. 현재 시뮬 조건(에스컬 실효처리율 0.6 ped/s, 2개)에선 에스컬이 여유 있어 최적 배합 선택에 영향을 주진 않음.
2. 하지만 정성적 병목 전이는 관측됨 → **에스컬 용량이 더 작거나 유입이 더 많으면 최적 배합이 달라질 가능성 시사**.
3. 후속 연구: (a) 에스컬 용량 sensitivity scan, (b) 비최적 config 1~2로 고의 제한하여 에스컬 완충 효과 관찰, (c) 열차 3편 이상(SIM_TIME=450s+)로 포화 상태에서 재검증.
