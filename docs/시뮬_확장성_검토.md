### 1-2. 현재 구현 범위

**입력**:
- 도착 모델: 물리 기반 (승강장 105m 위 하차 → 지연 → 계단 하행 12s
  → 대합실 진입)
- 열차 간격 180s, 편당 하차 234명 (서쪽 50%)

**공간**:
```
[승강장]   → 구현 안 됨 (시간 기반 생성 포인트로 축약)
    ↓
[계단 2개] → STAIRS (upper y=15~18, lower y=8~11), x=1
    ↓ 자유보행 (CFSM V2)
[대합실]  → 50m × 25m, 상단 L자형 들여쓰기 (NOTCH_X=12, NOTCH_Y=22)
    ↓
[게이트 7대] → x=12, 소프트웨어 큐 + LRP 선택 + 서비스 시간 모델
    ↓
[무료 대합실] → 자유보행 (post_gate waypoint)
    ↓
[출구 upper/lower] → add_exit_stage Polygon (현재 폭 1m로 수정됨, 2026-04-17)
    ↓
[에이전트 소멸]
```

**모델**:
- **물리**: CFSM V2 (JuPedSim CollisionFreeSpeedModelV2) — 속도 기반
  충돌 방지, Tordeux et al. (2016)
- **게이트 선택**: Gao et al. (2019) LRP + Y_WEIGHT 2.5 + 예측 큐 (FOV 8m)
- **큐**: 순수 Python FIFO + ease-in 진입 + 동적 waypoint (양방향 갱신)
- **서비스 시간**: 태그 lognormal(μ=2.0s), 태그리스 1.2s 고정

**출력**:
- `trajectories.csv`: 에이전트 궤적 (id, t, x, y, ...)
- `queue_history_cfsm.png`, `service_time_cfsm.png`, `snapshots_cfsm.png`
- `simulation_cfsm.mp4`, `simulation_cfsm_view.mp4`

### 1-3. 확장 필요 사항 (재정립 RQ 기준)

| 항목 | 현재 | 재정립 필요 |
|------|------|------------|
| **승강장** | 없음 (시간 기반 생성만) | 도어 위치별 생성 포인트 × 열차 도착 이벤트 |
| **출구 병목** | `exit_stage`로 즉시 소멸. 2026-04-17에 폭 1m 축소 시도했으나 유출량 제한 없음 | 에스컬레이터 처리율 제한 (1.1~1.3 ped/s) + 물리 funnel 벽 |
| **통행비용 계산** | 없음 (개별 지표만) | PedPy 기반 구간별 밀도·LOS → 통합 비용 산출 |
| **다시나리오 러너** | 단일 실행 | 50 시나리오 × 10 반복 자동화 (scenario_runner 필요) |
| **구간별 지표** | 게이트 중심 (큐, 서비스시간, MD) | 계단/대합실/게이트/출구 모두 |

### 1-4. DO NOT REVERT 13개 항목 충돌 분석

`CLAUDE.md` 기록 13개 항목은 **모두 보행 행태 자연스러움 보장용**.
공간 확장(승강장/출구) 또는 통행비용 로직 추가 시 다음 원칙:

| 항목 | 확장 시 유의 |
|------|-------------|
| #1 QUEUE_RESELECT_ENABLED=False | 에스컬레이터 앞은 FIFO 아님 → 새 스테이지로 분리 |
| #2 Y_WEIGHT=2.5 | 게이트 선택용, 에스컬레이터 선택엔 별도 파라미터 필요 |
| #3~#7 큐 관련 | 에스컬레이터 큐가 아닌 **군집 대기** 모델링 → 별도 로직 |
| #8 자유보행 중 재선택 제거 | 출구 선택도 spawn 시 1회로 제한 권장 |
| #9~#10 예측 큐 + 역행 방지 | 게이트 로직 그대로 유지 |
| #11 동적 waypoint 양방향 | 에스컬레이터 앞도 동일 원칙 적용 가능 |
| #12 SIM_TIME=120 | **재정립 시 600s로 확장 필요** (열차 3편) |
| #13 VISION_RANGE=8m | 게이트 전용. 출구/에스컬레이터 선택에는 별도 값 검토 |

**결론**: 13개 항목은 본래 **게이트 구간**에만 적용되므로, 확장된 출구/승강장
로직과 직접 충돌하지 않음. 다만 SIM_TIME 확장은 필수.

## 2. 레퍼런스 레포 재평가

### 2-1. JuPedSim (PedestrianDynamics/jupedsim)

- 이미 본 연구의 핵심 라이브러리. 1.3.x 버전 사용 중.
- GitHub: https://github.com/PedestrianDynamics/jupedsim
- 공식 문서: https://pedestriandynamics.org/

**활용 가능 기능** (본 연구 확장 관점):

| 기능 | 활용 시나리오 | 현재 사용 여부 |
|------|--------------|----------------|
| `Journey` 네트워크 | 다단계 경로(계단→대합실→게이트→출구) 구성 | 사용 중 (gate_wp → post_gate → exit) |
| `add_waypoint`, `add_exit_stage` | 정적 경로 요소 | 사용 중 |
| `add_notifiable_queue_stage` | 게이트 큐 모델링 | **미사용** (자체 Python FIFO로 대체됨) |
| `add_waiting_set_stage` | **열차 도착 대기** (승강장) | 미사용 — 승강장 확장 시 유력 후보 |
| `Transition.create_fixed_transition` | 단일 목적지 전환 | 사용 중 |
| `Transition.create_round_robin_transition` | 라운드로빈 분배 | 미사용 — 게이트 배합 실험 시 참고 가능 |
| `Transition.create_least_targeted_transition` | 최저 큐 선택 | 미사용 — LRP와 비교 실험 시 |
| Geometry switching | 열차 도어 열림/닫힘 | **검증 필요** — 1.x 버전 API 확인 필요 |

**승강장 모델링 제안**
- `add_waiting_set_stage(positions=[door_1, door_2, ...])` 로
  "열차 대기" 상태 표현
- 열차 도착 이벤트 시 `notify(waiting_set, agent_ids)` 호출로
  waiting 해제 → 계단 방향 Journey 진행

### 2-2. transp-or/pedestrian-control-simulator (가치 재평가)

- EPFL TRANSP-OR 연구실 산출물로 추정. 다구간 교통허브 전체 모델링에
  특화된 시뮬레이터.
- **검증 필요**: 공개 레포 URL, 라이선스, 최신 업데이트 상태.

**차용할 수 있는 설계 개념** (JSON 시나리오 설정 기준):

| 개념 | 설명 | 본 연구 활용 |
|------|------|-------------|
| `flow_gates` | 게이트 운영 정책 (개폐, 방향) | 태그/태그리스/겸용 배합 표현 |
| `controlled_areas` | 밀도 제어 영역 | 대합실/출구 밀도 모니터링 |
| `binary_gates` | on/off 게이트 | 시간대별 가변 운영 시나리오 |
| `flow_separators` | 물리적 흐름 분리 | 태그/태그리스 전용 차선 개념화 |
| `moving_walkways` | 에스컬레이터/무빙워크 | 출구 에스컬레이터 모델링 |
| `alternate_graphs` | 상황별 route choice 전환 | 동적 운영 로직 (후속 연구) |
| `destination_groups` | 이용자 유형별 목적지 집합 | Choice Set 차별화 |
| `lausanne-current/`, `three-platforms/`, `piw-corridor/` 시나리오 | 승강장 포함 다구간 시뮬 설정 구조 | 참고 예제로 활용 |

**직접 적용 가능성**
- JuPedSim은 Python + CFSM 기반, transp-or 은 보고된 바로는 C++ 이나
  구체 정보는 검증 필요. **직접 코드 이식은 어려움**, **설계
  아이디어만 참고** 권장.
- 특히 `moving_walkways`와 `flow_separators` 개념은 본 연구의
  출구 모델링에 직접 영감을 줄 수 있음.

### 2-3. PedPy (PedestrianDynamics/PedPy)

- GitHub: https://github.com/PedestrianDynamics/PedPy
- JuPedSim 파트너 프로젝트. 궤적 CSV → 밀도/속도/흐름 계산.

**본 연구 활용 가능성 (**매우 높음**)**:

| 기능 | 본 연구 활용 |
|------|-------------|
| `compute_voronoi_density` | **통행비용 함수의 밀도 항 산출** — 격자 방식 대비 개인별 밀도 정밀 측정 |
| `compute_classic_density` | 구간별 평균 밀도 | LOS 등급 분류 |
| `compute_flow` | 구간 통과 단면 유량 | 계단/게이트/에스컬레이터 출력 검증 |
| `compute_individual_speed` | 보행자별 속도 | 속도-밀도 관계 (통행비용 후보 (c)) |
| `compute_fundamental_diagram` | 기본 다이어그램 | 구간별 병목 여부 판정 |

**연동 방식**
- JuPedSim `trajectories.csv` → PedPy `TrajectoryData` 객체 직접 로딩
- 측정 영역(measurement area)은 Shapely Polygon으로 지정
- 본 연구의 통행비용 계산 파이프라인: `trajectories.csv` → PedPy로
  구간별 밀도 시계열 → 개인별 체류시간 × 밀도 적분 → TC_i

## 3. 확장 로드맵 제안

### 3-1. 단기 (중간발표 2026-04-20까지)

**목표**: 재정립된 연구 설계 논리를 시각적으로 시연할 최소 확장.

1. **출구 병목 최소 모델 시연용 구현**
   - 처리율 제한 로직 한 함수로 구현 (매 dt마다 exit 통과 N명 제한)
   - 파라미터: 에스컬레이터 1.1 ped/s × 2개 = 2.2 ped/s 총 유출
2. **파일럿 시나리오 2개 실행**
   - B0 (현행 겸용) vs B2 (순수 분리)
   - p=50% 고정, 첨두 시간대 1회씩
3. **PedPy로 구간별 밀도 측정 파이프라인 프로토타입**
   - 대합실(유료/무료), 게이트 구간, 출구 병목 앞

### 3-2. 중기 (5월)

## 5. 중간발표 전 최소 실행 가능 확장안

- **핵심**: 출구 유출량 제한 함수 1개 + B0/B2 파일럿 실행
- **작업량 추정**: 0.5일 (함수 구현) + 0.5일 (시나리오 실행 및 그래프)
- **시연 결과물**:
  - B0 vs B2의 게이트 큐 vs 출구 앞 밀도 비교 그래프
  - 병목 전이가 **실제로** 발생함을 시각적으로 입증
  - RQ3 검증 가능성 demonstrate

본 최소 확장은 **DO NOT REVERT 13개 항목 전체를 보존**하며, 출구 구간
로직만 별도 함수로 추가하므로 기존 검증된 보행 행태에 영향 없음.
